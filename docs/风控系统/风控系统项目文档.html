一、项目概述

风控系统是一套面向博彩/风控场景的用户行为特征工程 + 团伙可解释性系统。 系统由两大核心服务组成：risk_feature（特征工程 + 团伙可解释性）和 model_service（离线训练 + 消费推理聚类）。 通过实时与离线双链路协同，实现从原始用户行为数据到团伙聚类、风险告警的全链路处理。

1.1 项目目标

  • 实时特征计算：每 5 分钟增量拉取 StarRocks 数据，写入 Redis 热缓存，对活跃用户在 6h 窗口内重算特征
  • 离线特征生成：批量生成历史 6h 快照供模型训练（120 windows x 6h）
  • 可解释性产品：生成 summary_features、user_paths、sankey_edges 解释表
  • 团伙摘要：聚类完成后通过 MQ Consumer 预计算团伙/用户摘要、风险评分
  • 运营 API：FastAPI 暴露桑基图、摘要、PCA-3D、风险告警、平台产品列表

2.2 双链路设计

A 实时链路 (Real-time Pipeline)
  • 触发频率：每 5 分钟
  • 数据拉取：增量从 StarRocks 拉取最新 2min shard，写入 Redis（MessagePack 编码）
  • 特征计算：对 6h 窗口内的活跃用户重新计算全量特征
  • 输出：Parquet 文件 + explain 产出 (summary_features, user_paths, sankey_edges)
  • 后续：通过 RocketMQ 发送 feature_generated 消息通知 model_service
B 离线链路 (Offline Pipeline)
  • 触发方式：手动脚本 (gen_features_offline)
  • 生成范围：120 个历史窗口 x 6h 快照
  • 用途：供模型训练使用的标准化特征数据集
  • 输出目录：risk_data/offline/

四、数据流设计

4.1 实时链路数据流

StarRocks DWD → Redis 2min Shards → Feature Worker (5min) → Parquet + Explain → RocketMQ → Model Consumer → HDBSCAN → RocketMQ → Cluster Consumer → PostgreSQL Alerts

Redis 缓存策略：每条用户事件以 MessagePack 编码，按 2 分钟分片存储。6h lookback 窗口意味着最多缓存 180 个分片/用户。Worker 每 5 分钟拉取增量数据，仅重算活跃用户的特征。

4.2 离线链路数据流

StarRocks 历史数据 → gen_features_offline → 120 x 6h Snapshots → Parquet (training set) → Model Training

5.1 时间窗口设计

回看窗口
6 小时 (Lookback)
子窗口宽度
30 分钟 (Subwindow)
滑动步长
10 分钟 (Stride)
子窗口数量
34 个 (Subwindows)
Top-K 游戏槽位
5 个 Game Slots
实时触发间隔
5 分钟

窗口计算逻辑：6h = 360min，子窗口 30min，步长 10min。从窗口起始每 10min 移动一次子窗口起点，共 (360 - 30) / 10 + 1 = 34 个子窗口。每个子窗口覆盖 30min 内的用户行为事件。

5.3 特征分组

分组 列数 (约) 说明
P0 Masks ~20 子窗口活跃掩码、数据质量标记

Behavior ~600+ 序列特征 (seq) + 聚合特征 (agg) + 上下文哈希 (ctx_hash)

5.4 用户路径 (User Paths)

1 user_paths (单窗口路径)

记录用户在单个 6h 窗口内的行为 token 链。每个 token 编码了用户的关键行为节点（如登录、投注、提现、游戏切换），形成时序行为序列。

2 user_paths_full (跨窗口拼接)

跨快照拼接，最多回溯 30 个历史窗口，构建用户长期行为轨迹。用于捕捉跨天、跨周期的行为模式演变。

六、模型架构

6.1 GangModel 总览

GangModel 采用 Mamba 编码器 + KAN 投影头 的自监督架构，通过对比学习（InfoNCE + VICReg）学习用户行为嵌入表示，随后使用 HDBSCAN 进行团伙聚类。

┌──────────────────────────────────────────────────────────────────────┐ │ GangModel Architecture │ ├──────────────────────────────────────────────────────────────────────┤ │ │ │ Input Features (3329 cols) │ │ │ │ │ ├── Discrete Features ──▶ DiscreteEmbedder │ │ │ macro_path: 5 slots × 34 cols → 5×16 = 80 │ │ │ behav embeddings → 24 │ │ │ Total: [B, T, 104] │ │ │ │ │ └── Continuous Features ──▶ ContProjector │ │ Linear(2936 → 256) + LayerNorm │ │ mask_subwin applied │ │ Output: [B, T, 256] │ │ │ │ Fuse Layer │ │ Linear(360 → 256) + LayerNorm + pos_embed │ │ Input: cat([B,T,104], [B,T,256]) = [B, T, 360] │ │ Output: [B, T, 256] │ │ │ │ Mamba Encoder │ │ 4 layers, d_state=16, expand=2 (d_inner=512) │ │ Masked mean pooling → h [B, 256] │ │ │ │ ┌──────────────────────┐ ┌─────────────────────────┐ │ │ │ KAN Head │ │ Risk Score Head │ │ │ │ Spline KAN │ │ Linear(256→1) │ │ │ │ 256→128→64 │ │ + Sigmoid │ │ │ │ → z [B, 64] │ │ → s [B, 1] │ │ │ └──────────────────────┘ └─────────────────────────┘ │ │ │ └──────────────────────────────────────────────────────────────────────┘

6.2 组件详解

1 DiscreteEmbedder
  • macro_path：5 个 Top-K 游戏槽位 x 34 个离散特征列 → 每组通过 Embedding(vocab, 16) 映射 → 5 x 16 = 80 维
  • behav embeddings：行为类别特征嵌入 → 24 维
  • 输出：[Batch, T=34, 104]
2 ContProjector
  • 输入：2936 维连续特征（扣除 Key cols 和离散特征后的连续列）
  • 结构：Linear(2936 → 256) + LayerNorm
  • mask_subwin：根据 P0 mask 屏蔽无效子窗口
  • 输出：[Batch, T=34, 256]
3 Fuse Layer
  • 拼接：cat(DiscreteEmbedder output, ContProjector output) → [B, T, 360]
  • 投影：Linear(360 → 256) + LayerNorm
  • 位置编码：可学习 positional embedding 加入
  • 输出：[Batch, T=34, 256]
4 Mamba Encoder
  • 层数：4 layers
  • 状态维度：d_state = 16
  • 扩展因子：expand = 2 (d_inner = 512)
  • 池化：Masked mean pooling (排除无效子窗口)
  • 输出：h [Batch, 256] — 用户全局表示向量
5 KAN Head (Spline KAN)
  • 结构：256 → 128 → 64 (Spline basis functions)
  • 输出：z [Batch, 64] — 用于聚类的低维嵌入
  • 用途：HDBSCAN 对 z 向量进行团伙聚类
6 Risk Score Head
  • 结构：Linear(256 → 1) + Sigmoid
  • 输出：s [Batch, 1] — 个体风险概率分数

6.3 训练策略

参数 值 说明
训练方式 SSL (自监督学习) 无需人工标注
对比学习 Two Augmented Views 对同一样本生成两个增强视图
损失函数 InfoNCE + VICReg 对比损失 + 方差-不变性-协方差正则
Batch Size 4096 大批量对比学习
评估指标 Jaccard + Embedding Stability 聚类稳定性评估

InfoNCE：最大化同一用户不同增强视图的相似度，最小化不同用户间的相似度。
VICReg：Variance（保持特征方差）+ Invariance（增强不变性）+ Covariance（去相关性），防止表示坍缩。

七、团伙可解释性

7.1 聚类摘要

1 cluster_summary (聚类层摘要)

每个聚类簇的统计概览：成员数量、平均风险分数、主要行为模式、核心游戏偏好、提现特征分布。

2 user_summary (用户层摘要)

单个用户在其所属团伙中的画像：个体风险分数、与团伙均值偏差、行为路径特征、关键异常指标。

3 summary_features (摘要特征表)

聚合后的解释性特征表，包含团伙级和用户级的关键指标切片，供运营人员直观理解。

7.2 团伙风险评分

对聚类中的提现成员子集进行线性加权评分，范围 0-100。

指标 权重 说明
withdraw_ratio_in_cluster 35 聚类内提现成员占比
path_top1_withdraw 15 最频繁行为路径中提现占比
rhythm_top1 10 操作节奏异常度 (Top-1)
volatility_trend_top1 10 波动趋势异常度 (Top-1)
withdraw_gt_stake 10 提现金额超过投注金额的比率
game_switch_top1 10 游戏切换频率异常度
agent_channel_top1 10 代理渠道集中度

风险等级划分：

Critical (≥80)  High (≥60)  Medium (≥40)  Low (<40)

7.3 桑基图 (Sankey)

  • Global Sankey：全局用户行为流转图，展示从不同行为节点到最终状态的流量分布
  • Simple Sankey（6 类粗粒度）：将行为归类为 6 大类别的简化流转图
  • Cluster Sankey：按聚类 ID 筛选，展示特定团伙内的行为流转模式

十三、容错与监控

13.1 容错机制

1 重试机制 (Retry)
  • StarRocks 查询：指数退避重试，最大 3 次，超时 30s/次
  • Redis 操作：连接池 + 自动重连，写入失败静默跳过（不影响主流程）
  • RocketMQ：消费失败自动 requeue，最大重试 5 次后进入死信队列
  • PostgreSQL：连接池管理，事务级重试
2 降级策略 (Fallback)
  • Redis 不可用：Worker 降级为直接从 StarRocks 全量拉取（性能下降但不停服）
  • MQ 不可用：特征文件写入本地，待 MQ 恢复后通过 mq_producer 脚本补发
  • 模型推理超时：返回上一次有效聚类结果
3 水位线 (Watermark)
  • Worker Watermark：记录最后成功处理的 as_of_ts，重启后从水位线继续
  • Consumer Offset：RocketMQ consumer offset 持久化，确保不丢消息
  • 数据完整性：Parquet 写入采用原子重命名（write tmp + rename），避免读取不完整文件

13.2 监控与日志

监控维度 实现方式 告警条件
Worker 心跳 Systemd watchdog + 日志时间戳 超过 15min 无新日志
特征延迟 as_of_ts 与当前时间差 延迟超过 30min
聚类质量 metrics.json 中 silhouette_score 得分低于 0.2
MQ 消费延迟 Consumer lag 监控 积压超过 10 条
API 健康 /healthz + /readyz 端点 连续 3 次检查失败
磁盘使用 risk_data 目录大小监控 使用率超过 80%

十四、运维指南

14.1 冷启动流程

场景：首次部署或 Redis 数据丢失后，需要重建缓存和特征基线。

  1. 执行冷启动脚本：python -m scripts.cold_start
    • 从 StarRocks 拉取最近 6h 的全量事件数据
    • 写入 Redis 缓存（按 2min 分片）
    • 预热完成后 Worker 可正常增量运行
  2. 验证数据完整性：python -m scripts.verify
    • 检查 Redis 中各 platform-product 的分片覆盖率
    • 校验特征列数是否符合 Schema v2.14 (3334 列)
  3. 启动 Worker：systemctl start risk_feature_worker
  4. 启动 Consumer：systemctl start risk_feature_cluster_consumer

14.2 离线训练流程

  1. 生成离线特征：
    python -m scripts.gen_features_offline \ --start-date 2026-04-01 \ --end-date 2026-05-01 \ --platform platform_a \ --product slots
  2. 执行模型训练：
    python -m scripts.train \ --data-dir /home/ai-local2/risk_data/offline/features/ \ --epochs 100 \ --batch-size 4096
  3. 评估模型：
    python -m scripts.evaluate \ --checkpoint /home/ai-local2/model_service/checkpoints/latest.pt
  4. 执行聚类测试：
    python -m scripts.cluster \ --checkpoint /home/ai-local2/model_service/checkpoints/latest.pt \ --feature-file /home/ai-local2/risk_data/online/features/latest.parquet

14.3 常见故障排查

故障现象 可能原因 排查步骤
Worker 无输出 StarRocks 连接失败 / Redis 不可达 检查网络连通性，查看 worker 日志中的连接错误
特征列数不匹配 Schema 版本不一致 确认 schema_registry.py 版本，运行 verify 脚本
聚类结果全为噪声(-1) HDBSCAN 参数不适配 / 数据量不足 检查 min_cluster_size，确认输入数据行数 > 100
MQ 消费停滞 Consumer 进程崩溃 / 死信积压 检查 systemd 状态，查看 RocketMQ 控制台 consumer lag
API 响应慢 大量 Parquet 文件扫描 / PG 查询慢 检查 products 目录文件数，优化 PG 索引
Redis 内存持续增长 TTL 未设置 / 过期分片未清理 检查 Redis INFO memory，确认分片 TTL 配置
风险告警未触发 cluster_consumer 未运行 / PG 连接失败 检查 consumer 日志，验证 PG 表写入权限
